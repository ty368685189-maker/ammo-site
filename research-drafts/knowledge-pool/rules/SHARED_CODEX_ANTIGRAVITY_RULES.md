# Codex 与 Antigravity 统一知识库规则

这个文件是统一口径。以后 Codex 和 Antigravity 扩充知识库，都必须按这里执行。

## 1. 目标一致

Codex 和 Antigravity 做的是同一种任务：

```text
本地入库预备级知识库扩充
```

不是随便聊天列候选，也不是正式入库。

结果必须做到：

- 同一个保存目录。
- 同一个目录结构。
- 同一个字段模板。
- 同一个搜索深度。
- 同一个证据强弱口径。
- 同一个汇报格式。
- 同一个“不入库”边界。

## 2. 固定保存位置

所有知识库成果都保存到：

```text
E:\弹药网站\ammo-site\research-drafts\knowledge-pool\
```

目录结构固定为：

```text
knowledge-pool/
  INDEX.md
  FORMAL_LIBRARY_SNAPSHOT.md
  rules/
  batches/
    mixed/
    enterprise/
    university/
  raw-json/
  notes/
```

不能保存到：

- 聊天框临时输出。
- Antigravity 工作区。
- 桌面。
- 下载目录。
- 其他临时目录。

## 3. 每轮必须产出

每轮正式知识库任务必须产出：

- 本轮 Markdown 文件，保存到 `batches/` 对应子目录。
- 推荐配套 `.raw.json`，保存到 `raw-json/`。
- 更新 `INDEX.md`。

只要宇少说“做知识库 / 扩知识库 / 继续知识库”，就必须落文件。

下面这些一句话，也都按“正式开始扩知识库”处理，不需要宇少再补任务分配：

- `继续扩知识库`
- `做一轮知识库`
- `继续补企业知识库`
- `继续补院校知识库`
- `先扩一点知识库`

默认规则：

- 没说类型时，优先企业知识库扩充。
- 说了企业，就做企业批次。
- 说了院校，就做院校批次。
- 没说“测试 / 只看效果 / 不写文件”时，必须落文件并更新 `INDEX.md`。

只有宇少明确说“测试 / 只看效果 / 不写文件”，才可以只在聊天里输出。

## 4. 质量目标

知识库资料要达到“入库预备级”。

意思是：

- 后续能较快转成 clean JSON。
- 不需要重新大搜。
- 来源链接可追溯。
- 证据强弱清楚。
- 风险和待核对字段清楚。
- 不把待核对内容写成确定事实。

但知识库阶段不要正式入库。

## 5. 统一字段模板

统一使用：

```text
E:\弹药网站\ammo-site\research-drafts\knowledge-pool\rules\KNOWLEDGE_POOL_ENTRY_TEMPLATE.md
```

企业和院校字段必须按这个模板尽量补齐。

缺字段可以写：

- 未知。
- 待核对。
- 风险。

不能空着不说明。

## 6. 统一质量标准

统一使用：

```text
E:\弹药网站\ammo-site\research-drafts\knowledge-pool\rules\IMPORT_READY_KNOWLEDGE_STANDARD.md
```

基本要求：

- 每条至少 1 个可打开关键入口。
- 优先 2 个以上来源。
- 强候选建议至少 3 个来源。
- 只有 1 个来源时，证据强弱最多写“中”，不能写“强”。
- 官网打不开或只有招聘平台/第三方转载时，写“中 / 待核对”。
- 弱来源不能单独支撑核心事实。

企业条目补充硬规则：

- `招聘入口 / 招聘公告` 不能随便回填官网首页；没有独立招聘页就写 `待核对`。
- `关键入口` 和 `来源记录` 里的 URL，必须尽量指向对应具体页面，不能用同一个首页 URL 冒充多个来源。
- 企业来源类型不要写院校口径，例如不要写 `学校官网 / 招生目录`。
- 薪资、作息、福利没有强/中证据时，分别写 `未知`、`待核对`，不要写死成 `双休`、`五险一金` 等确定字段。
- `searchTrace` 里提到的结论，如果没有对应 `sources` 链接支撑，就只能写成缺口或待核对，不能写成已确认发现。

这几条是审查红线。踩中任意一条，条目最多算“基本合格，需修后再算合格”。

## 7. 统一搜索方式

每轮先广度搜索，再单条多轮核验。

企业至少搜：

1. 主体名称 + 官网。
2. 主体名称 + 招聘 / 校园招聘 / 学校就业网。
3. 主体名称 + 方向词。

院校至少搜：

1. 学校名称 + 招生网 / 本科招生。
2. 学校名称 + 学院 / 专业介绍。
3. 学校名称 + 研究生招生目录 + 方向词。

每条候选必须记录检索关键词和检索轮次。

## 8. 统一避重方式

每轮开始先读：

```text
FORMAL_LIBRARY_SNAPSHOT.md
INDEX.md
最近 3-5 个 `batches/` 里的知识库批次文件
```

必须标注：

- 是否正式库重复。
- 是否知识库重复。
- 是否集团/子公司强重叠。
- 是否旧名或重组前主体。
- 院校是否同名或相近院校混淆。

## 9. 统一边界

知识库阶段不要：

- 不改 `src/data/companies.ts`。
- 不改 `src/data/universities.ts`。
- 不运行 `npm run import:company`。
- 不运行 `npm run import:university`。
- 不改页面。
- 不改脚本。
- 不改部署。
- 不碰服务器、VPS、域名。

## 10. 统一汇报格式

每轮结束统一汇报：

```text
宇少，本轮知识库搜集完成。

- 新增文件：
- raw JSON：
- INDEX 是否更新：
- 新增候选：
- 企业：
- 院校：
- 较强候选：
- 暂缓候选：
- 没有改正式库：
- 没有改页面：
- 没有部署 / VPS / 域名操作：
- 下一步建议：
```

## 11. 谁来做都一样

Codex 做，按这个文件。

Antigravity 做，也按这个文件。

如果其他会话、其他工具接手，也按这个文件。

## 12. 审查问题必须沉淀

每次知识库审查后，如果结论是：

- 基本合格需修
- 不合格

必须把问题追加到：

```text
rules/KNOWLEDGE_POOL_REVIEW_LESSONS.md
```

记录内容至少包括：

- 日期。
- 来源：Codex / Antigravity / 其他。
- 文件路径。
- 审查结论。
- 主要问题。
- 应沉淀规则。
- 后续检查口径。

目的：把每次踩坑变成后续规则，避免同类错误反复出现。
