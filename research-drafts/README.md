# research-drafts 草稿与证据流程

这个目录放两类内容：知识库候选池和待审核入库草稿，都不是正式数据库。

- `knowledge-pool/`：企业和院校候选信息池，先攒名称、地区、方向、官网/招生/招聘入口、证据强弱、重复风险和建议动作。当前阶段可以数量优先，正式入库后置。
- `companies/`、`universities/`：已经准备进入正式库的 clean 草稿，需要 dry-run、评分和数据质量检查。

当前不优先使用 Grok。Grok 搜索和整理不稳定，除非宇少明确要求，否则由 Codex 或 Antigravity 直接做知识库候选搜集；正式入库仍由 Codex/Antigravity 审证据、改成干净 JSON、跑 `--dry-run`、给质量评分。

评分细则见 [SCORECARD.md](SCORECARD.md)。常用命令：

```bash
npm run score:drafts
npm run score:drafts -- "research-drafts/companies/草稿.json"
```

## 1. 证据包接收标准

### 强证据

- 企业：企业官网、集团官网、官方招聘公告、学校就业网原始招聘简章、政府部门官网、信用中国或监管部门正式文书。
- 院校：学校官网、本科招生网、学院官网、官方招生章程/招生计划/招生目录、学校官方就业质量报告、本科教学质量报告。
- 可支撑字段：主体名称、官网入口、学院/专业入口、招生专业、招聘岗位、官方公告里的工作地点、官方公告里的薪资和学历要求。
- 使用要求：记录标题、类型、日期和 URL；核心结论最好至少有 1 条强证据支撑。

### 中证据

- 企业：权威媒体、集团新闻、学校就业网转载、招聘平台公司主页、官方社媒历史内容。
- 院校：学校二级单位新闻、学院通知、研究生招生网、教学管理部门页面、权威平台转载的官方信息。
- 可支撑字段：业务方向、历史招聘线索、学院方向、培养方向、专业相关性线索。
- 使用要求：只能写成“线索”“历史样本”“待核对”，不能扩写成当前稳定制度。

### 弱证据

- 百度百科、工商聚合页、第三方院校库、招聘平台评论、论坛问答、自媒体文章、来源不明 PDF 镜像。
- 可支撑字段：一般只能作为检索线索或补充来源。
- 使用要求：不得单独支撑薪资、作息、就业率、分数线、招生人数、处罚记录、荣誉资质等核心字段。

### 不可用证据

- 无 URL、打不开、标题与页面内容不符、明显复制站、疑似 AI 汇总、域名混淆、和目标主体不是同一单位。
- 处理方式：不写入 `links` 或 `research.sources`；如有混淆风险，可以放进 `research.searchTrace.gaps` 提醒后续避开。

## 2. 企业证据包核对模板

接到企业证据包后，先按下面顺序审，不急着写 JSON。

| 核对项 | 优先来源 | 写入规则 |
| --- | --- | --- |
| 主体名称 | 官网、集团页、政府资料、官方招聘公告 | 名称不一致时保留风险说明，不能混用子公司和总部 |
| 官网/招聘入口 | 官网、官方招聘页、学校就业网、国聘等平台 | 官网可进 `website`；招聘平台只能算入口或样本 |
| 业务方向 | 官网、集团资料、年报/公告、政府资料 | 可写行业标签；弱来源只能写“待核对” |
| 薪资 | 官方招聘公告、学校就业网原始简章 | 无强/中证据写 `未知`；历史样本写 `样本值（待核对）` |
| 作息 | 官方招聘公告、劳动合同说明、可靠招聘公告 | 无来源写 `待核对`，不要凭平台福利标签判断 |
| 岗位 | 官方招聘公告、学校就业网、招聘入口 | 写“历史岗位样本”时要标年份或待核对 |
| 学历 | 官方招聘公告、学校就业网 | 不确定写 `待核对` |
| 工作地点 | 官方招聘公告、岗位详情 | 多地点或项目制要写风险，不要只写总部城市 |
| 待核对字段 | 最新校招、签约主体、薪资口径、作息、福利、项目派驻 | 放进 `verification.pendingFields` 和 `research.risks` |

企业草稿最低自检：

- `description` 不超过 40 字。
- `ownership` 只能是 `国企` / `民企` / `外企` / `创业公司` / `待核对`。
- `schedule` 只能是 `双休` / `大小周` / `单休` / `待核对`。
- `research.sources` 必须存在，尽量 5 条以上。
- `research.conclusion` 控制在 80 到 160 字。

## 3. 院校证据包核对模板

院校最容易把“研究生方向、学院科研平台、第三方院校库”误写成本科招生事实，必须逐项核对。

| 核对项 | 优先来源 | 写入规则 |
| --- | --- | --- |
| 学校名称/简称 | 学校官网、本科招生网 | 简称用常见中文简称，不写中英混杂 |
| 本科招生专业 | 本科招生网、招生章程、招生计划、专业介绍 | 不得用研究生目录或科研平台替代本科招生证据 |
| 学院入口 | 学校官网导航、学院官网 | 核对域名，防止同省高校混淆 |
| 招生目录/计划 | 本科招生网、官方招生章程、分省计划 | 分数线、人数、选科要求必须按年份和省份复查 |
| 就业数据 | 学校官方就业质量报告、学校就业信息网 | 第三方 PDF 镜像不能支撑强结论 |
| 层次标签 | 教育部/学校官方信息、学校官网 | 双一流但非 211：`tier` 写 `普通本科`，`tags` 写 `双一流` |
| 域名混淆风险 | 官网域名、学校二级域名、搜索结果标题 | 混淆域名写入 `searchTrace.gaps`，不要写入来源 |
| 待核对字段 | 当年专业、分省计划、录取分数、就业去向、专业名称变化 | 放进 `verification.pendingFields` 和 `research.risks` |

院校草稿最低自检：

- `overview` 不超过 60 字。
- `focus` 不超过 35 字。
- `tier` 只能是 `985 / 211` / `211` / `普通本科`。
- `research.sources` 必须存在，尽量 6 条以上。
- 不把研究生方向、科研平台、学院新闻写成本科招生专业。

## 4. 草稿 JSON 通用自检清单

保存到 `research-drafts/companies/` 或 `research-drafts/universities/` 前，逐项过一遍：

- JSON 能被解析，不能有重复字段。
- `kind`、`target`、`base`、`verification`、`research` 结构完整。
- 枚举值符合 `DATA_SCHEMA.md`。
- `links` 和 `research.sources` 不能有空 URL、假 URL、重复 URL。
- 来源类型不要抬高证据等级，弱来源优先标 `其他`。
- 薪资、作息、就业率、分数线、招生人数、荣誉、处罚记录没有强/中证据时，降级为 `待核对`、`未知` 或删除。
- 文案不要写“显著影响力”“就业竞争力强”“顶级平台”等宣传判断。
- 有不确定内容时，必须同步写进 `pendingFields` 或 `risks`。
- 检查是否把待核对内容写成确定事实。

## 5. dry-run 验证

只预览新增：

```bash
npm run import:company -- "research-drafts/companies/草稿.json" --dry-run
npm run import:university -- "research-drafts/universities/草稿.json" --dry-run
```

只预览更新已有条目：

```bash
npm run import:company -- "research-drafts/companies/草稿.json" --dry-run --update
npm run import:university -- "research-drafts/universities/草稿.json" --dry-run --update
```

正式导入前必须满足：

- dry-run 输出符合字段和文案要求。
- 质量评分和问题清单已经给宇少看过。
- 宇少明确说“确认正式导入”或“确认更新”。
