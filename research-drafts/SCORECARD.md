# 草稿质量评分清单

这个清单用于判断 `research-drafts/*/*.json` 是否可以进入正式导入流程。当前阶段宇少已确认：不做人工逐条核对，评分达标即可入库；人工核对后置。

## 命令

评分全部草稿：

```bash
npm run score:drafts
```

只评分某一个草稿：

```bash
npm run score:drafts -- "research-drafts/companies/草稿.json"
```

只看 2026-06-07 之后改过的草稿：

```bash
npm run score:drafts -- --since=2026-06-07
```

## 分数含义

- `85-100`：可正式导入，前提是 dry-run 通过且没有阻塞项。人工打开来源核对后置。
- `70-84`：先修问题再导入。通常是来源数、文案长度、待核对边界不够稳。
- `0-69`：不建议导入。先回到证据包重做。
- 有“阻塞”项：不建议导入，即使分数看起来不低。

## 阻塞项

- JSON 解析失败或有重复字段。
- 缺 `base`、`verification`、`research` 等核心结构。
- 枚举值不符合 `DATA_SCHEMA.md`。
- `links` 或 `sources` 有空 URL。
- 没有任何可追溯来源。
- dry-run 失败。
- 院校把 `双一流` 写进 `tier`，或安徽理工大学混入 `ahut.edu.cn`。

## 较重扣分项

- `description`、`overview`、`focus`、`research.conclusion` 超出项目限制。
- 企业来源少于 5 条，院校来源少于 6 条。
- `sources` 放在顶层而不是 `research.sources`。
- 使用百度百科、工商聚合页、第三方院校库等弱来源支撑核心字段。
- 薪资、作息、就业率、录取分数、招生人数写得太确定，但没有强/中证据。

## 轻微扣分项

- `pendingFields` 太少。
- `keySignals`、`suitableFor`、`risks` 数量不合适。
- 文案出现“全面、赋能、显著、顶级、竞争力强”等宣传味词。
- 作息等字段写成确定值时，仍需人工确认来源是否直接支撑。

## 后续人工核对重点

当前阶段评分达标可以先正式导入。后续宇少有时间时，再优先人工看：

- 来源标题、URL、主体名称是否一致。
- 来源是否属于目标企业/院校，而不是集团、子公司或同名单位混淆。
- 招生、薪资、作息、就业率、分数线、招生人数是否有年份和口径。
- 弱证据是否只作为检索线索，没有支撑核心字段。
- 文案是否克制，是否保留“待核对”边界。
